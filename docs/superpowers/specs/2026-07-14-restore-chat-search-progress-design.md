# 恢复日常聊天与搜索过程显示设计

## 目标

以 VPS 上 2026-06-04 的稳定备份为交互基线，恢复日常聊天、搜索进度和可查看执行链；保留当前的环境变量密钥、非 root systemd、接口鉴权、长期记忆接入和日报去重。

## 行为边界

- 普通聊天继续使用原来的自然中文正文，不添加“关键依据”等结构化包装。
- 每条 Telegram 对话先显示路线：纯模型、搜索、本地工具或读取日报。
- 搜索路线复原一条可编辑的进度消息，依次展示查询改写、采集、素材整理、正文生成、事实核查和发送状态。
- 搜索与抓取工具调用临时显示具体关键词或网址，最多保留三条，回答完成后清理。
- `/thinking` 展示最近一轮的路线、逐轮工具、搜索词、素材数、写作和核查结果；不输出模型逐字内部推理。
- `/tools` 继续提供最近工具记录；`/sources` 继续提供来源存档。
- HTTP `/ask` 不发送 Telegram 进度消息，也不污染 Telegram 对话状态。
- 日报继续使用已经恢复的中文两段式模板；部署完成后手动重新生成并推送一份。

## 组件

- `tg_bot/progress.py`：Telegram 进度消息控制器，只负责创建、编辑、临时工具状态和收尾。
- `tg_bot/pipeline/gather.py`：把每次实际工具调用交给可选状态回调。
- `tg_bot/core/pipeline.py`：在查询、采集、整理、写作和核查阶段触发进度回调。
- `tg_bot/bot.py`：按路由启动进度控制器，搜索时传入回调，发送后收尾。
- `tg_bot/commands/info.py`：从思考记录和工具记录生成安全、详细的执行链。

## 失败处理

- Telegram 进度消息创建或编辑失败不得影响最终回答。
- 搜索 pipeline 降级时显示“切换备用方案”，并继续原有 fallback。
- 无来源或异常结束时停止动画并显示失败状态，不能遗留永久“搜索中”。

## 验收

- 快速聊天显示路线与已发送状态，但不伪装成搜索。
- 搜索聊天能看到工具关键词、抓取、写作、核查和已发送状态。
- `/thinking` 能区分快速回答和搜索回答，并逐轮列出工具，不泄露逐字内部推理。
- 全量单元测试通过，远程服务健康，未授权 `/ask` 仍返回 401。
- 手动日报任务成功，状态为 `fresh` 且 Telegram 推送成功。
